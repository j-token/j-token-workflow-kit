## 인지 부하론 개요

인지 부하론(Cognitive Load Theory)은 인간의 **작업기억(working memory)이 한 번에 저장하고 처리할 수 있는 정보량이 매우 제한적**이라는 사실에서 출발한다. 작업기억 한계를 초과하면 학습·이해·판단 성능이 급격히 떨어진다. 부하는 세 가지로 구분된다.

| 유형 | 정의 | 글쓰기 맥락 |
|---|---|---|
| **내재적 부하** | 대상 자체의 본질적 복잡도. 사전지식 보유 정도에 따라 상대적으로 결정 | 다루는 주제 자체의 복잡도 — 변경 규모, 알고리즘, 도메인 규칙 |
| **외재적 부하** | 자료 제시 방식·교수 전략에서 발생하는 부하. 설계로 줄일 수 있음 | 글의 작성 방식 — 제목·구조·잡음·시각자료·마크다운 형식 |
| **본유적 부하** | 학습 목표에 직접 관련된 정신적 노력. 적절 수준에서 발생하는 긍정적 부하 | 주제의 의도·트레이드오프를 이해하는 데 쓰는 사고 |

글쓰기의 목표는 단순하다.

1. **내재적 부하**는 본질적이므로 줄일 수 없지만 **분할**할 수 있다.
2. **외재적 부하**는 작성자가 제거할 수 있다. 제거가 최우선 과제다.
3. **본유적 부하**는 독자가 주제의 의도에 집중하도록 **유도**해야 한다.

## 글쓰기에서의 적용

### 내재적 부하 줄이기

#### 글 크기 가이드

- **권장 크기**: 한 글은 한 화면(스크롤 1~2회)에 골격이 잡혀야 한다. PR이라면 변경 줄수 300~500줄 이내가 권장.
- **상한선**: PR 1,000줄, plan 단계 15개, 본문 섹션 9개를 초과하면 분할을 고려한다. 독자가 골격을 작업기억에 담을 수 없다.
- **예외**: 자동 생성된 코드(스키마, 락 파일, 마이그레이션 등)나 부록은 줄수에서 제외 가능하지만, "이 영역은 리뷰 대상이 아니다"라고 명시한다.

#### 단일 관심사 원칙

글 하나는 **하나의 관심사**만 다룬다. 다음은 모두 다른 관심사다.

- 기능 추가
- 버그 수정
- 리팩토링
- 포매팅·린트
- 의존성 업그레이드
- 변수·파일 리네이밍
- 설계 결정
- 실행 계획

기능 추가 중에 발견한 리팩토링은 **다음 글**로 미루거나, 별도 단위로 분리해 "이 단락은 동작 변경이 없습니다. 먼저 보시면 본 변경 의도가 명확해집니다"처럼 명시한다.

#### 단계적 글 체인

큰 변경·큰 주제를 단계적 글 체인으로 쪼개는 패턴(PR 기준).

1. **글 1**: 인터페이스·타입 정의(작은 글, 동작 변경 없음)
2. **글 2**: 구현부 추가(글 1에 의존)
3. **글 3**: 호출부 마이그레이션(글 2에 의존)
4. **글 4**: 구버전 제거(글 3에 의존)

각 글 본문에 의존 관계를 "관련 글" 섹션으로 명시해 독자가 컨텍스트를 잃지 않도록 한다.

### 외재적 부하 줄이기

#### 제목 패턴

좋은 제목은 **다음 세 가지를 1줄에 담는다**.

- **무엇이** 바뀌었는지 / 다뤄지는지
- **어떤 영역**에서 바뀌었는지
- **왜** 바뀌었는지(가능하면)

| Bad | Good |
|---|---|
| `feat: 수정` | `feat: 로그인 화면 토큰 만료 시 자동 재발급 추가` |
| `bug: 버튼 안 됨` | `bug: iOS 17에서 결제 버튼 더블 탭 시 중복 결제 방지` |
| `refactor: 정리` | `refactor: useAuth 훅을 zustand store로 이관(상태 동기화 단순화)` |
| `plan: 결제` | `plan: 결제 모듈 PaymentProvider 인터페이스 도입(1/4)` |

#### TL;DR 작성법

본문 최상단의 요약은 **3줄 이내**로 작성한다. 독자가 한 화면에서 골격을 잡을 수 있어야 한다.

```md
## 핵심 변경사항 요약
- 토큰 만료(401) 응답 시 자동 재발급 후 원 요청 재시도
- 재발급 동시 진행 방지를 위한 단일 promise 큐 도입
- 재발급 실패 시 로그아웃 처리 분기 추가
```

#### 변경 그룹화 — 순서가 독자의 이해 순서

PR의 커밋 분리든, plan의 단계 배치든, 문서의 섹션 순서든 **독자가 읽는 순서가 곧 이해 순서**가 되도록 배치한다.

- **잡음 단위를 먼저** — 포매팅·임포트 정리 같은 본질이 아닌 단위를 앞에 두고 "여기는 동작 변경 없음"이라고 명시.
- **인터페이스 → 구현 → 호출부** 순서로 단위를 배치.
- 단위 1개 = 독자가 한 번에 이해할 수 있는 덩어리.
- 분리가 불가능한 큰 글은 본문에 "진입점" 섹션을 추가해 가장 먼저 볼 위치를 지정.

#### 시각자료

UI·아키텍처·플로우는 텍스트만으로 인지 부하가 매우 크다. 다음을 첨부한다.

- **정적 UI**: Before/After 스크린샷
- **인터랙션**: 짧은 화면 녹화(5~15초)
- **다크모드/접근성**: 두 모드 모두 스크린샷
- **아키텍처/데이터 플로우**: Mermaid 다이어그램 또는 이미지

#### 잡음 분리

다음 변경은 별도 단위로 분리한다.

- 코드 포매팅(prettier·gofmt 적용)
- 임포트 정렬
- 사용하지 않는 변수 제거
- 변수·파일 리네이밍
- 주석 추가/제거
- 타입 어노테이션 보강(동작 변경 없음)

이들이 핵심 변경과 한 단위에 섞이면 독자가 "어디가 의미 있는 변경인지" 작업기억을 소모하게 된다.

### 본유적 부하 유도하기

#### 의사결정 로그 표기

"왜 이렇게 했는가" 섹션에는 **당연하지 않은 결정**만 적는다.

- 대안 A·B를 검토했다면 각 대안을 1줄로 적고, 선택한 이유를 명시한다.
- 외부 제약(API 한계, 라이브러리 버그, 정책)이 있다면 링크와 함께 적는다.
- 임시 처치(HACK)라면 명시하고, 추후 정리 시점·조건을 적는다.

#### 트레이드오프 명시

모든 결정에는 트레이드오프가 있다. 독자가 같은 추론을 반복하지 않도록 명시한다.

```md
## 왜 이렇게 했는가

토큰 재발급을 axios interceptor 안에서 처리하도록 했습니다.

- 대안 1: 각 API 호출부에서 401 처리 — 모든 호출부에 분기 코드가 필요해 누락 위험이 큼
- 대안 2: 상위 컴포넌트의 ErrorBoundary에서 처리 — 비동기 요청과 React 트리가 분리돼 있어 매핑이 어려움
- 채택안: interceptor — 단일 지점에서 모든 요청을 커버. 단점은 인터셉터 로직 자체가 복잡해진다는 점이며, 단위 테스트로 보완
```

#### 진입점(어디부터 볼지)

큰 글일수록 진입점을 명시한다. 독자가 파일 트리·목차를 보고 우왕좌왕하지 않도록 한다.

```md
## 진입점

1. `src/api/interceptor.ts:42-120` — 재발급 로직 핵심
2. `src/api/refresh-queue.ts` — 동시 재발급 방지 큐
3. `src/api/__tests__/interceptor.test.ts` — 핵심 케이스 검증
4. (그 외 파일은 호출부 마이그레이션)
```

#### 도메인 용어 보강

독자가 모를 수 있는 약어·도메인 용어는 1줄 설명을 단다.

```md
> **RTR(Refresh Token Rotation)**: 재발급 시마다 리프레시 토큰을 새로 발급하고 이전 토큰을 무효화하는 방식. 도난 시 피해 최소화 목적.
```

## Before / After 예시

### 예시 1: 모호한 제목 vs 의도 드러난 제목

**Before**

```text
bug: 결제 오류 수정
```

**After**

```text
bug: iOS 17 + 카카오페이 결제 시 웹뷰 콜백 누락으로 결제 미반영되는 회귀 수정
```

After가 더 길지만, 독자는 제목만으로 "어떤 환경에서, 어떤 결제 수단에서, 어떤 증상이, 어디서 발생했는지" 즉시 파악한다. 외재적 부하가 큰 폭으로 줄어든다.

### 예시 2: 거대한 단일 글 vs 분할된 글 체인

**Before** — `feat: 결제 모듈 리뉴얼` (2,300줄 변경, 47개 파일)

독자는 어디부터 볼지 모른다. 리뷰가 며칠 지연되고, 결국 "LGTM" 같은 형식 리뷰만 받게 된다.

**After**

- 글 1: `refactor: 결제 모듈 인터페이스(PaymentProvider) 정의` (120줄, 동작 변경 없음)
- 글 2: `feat: 카카오페이 PaymentProvider 구현` (450줄)
- 글 3: `feat: 네이버페이 PaymentProvider 구현` (380줄)
- 글 4: `refactor: 결제 호출부를 PaymentProvider로 마이그레이션` (620줄)
- 글 5: `chore: 구버전 결제 모듈 제거` (730줄, 대부분 삭제)

각 글은 독립적으로 빠르게 검토 가능하다. 독자의 작업기억이 한 글당 한 가지 책임만 다룬다.

### 예시 3: 변경 나열식 vs "왜→무엇→어떻게 검증" 구조

**Before**

```md
## 변경사항

- src/auth.ts 수정
- src/api.ts 수정
- 테스트 추가
```

독자는 코드 디프를 보고 "왜?"를 처음부터 추론해야 한다. 본유적 부하가 과도하게 발생한다.

**After**

```md
## 작업 목적

iOS 사용자가 백그라운드 진입 후 30분 이상 지나면 토큰 만료로 첫 API 요청이 실패하는 회귀 수정.

## 핵심 변경사항 요약

- 401 응답 시 자동 재발급 + 원 요청 재시도
- 재발급 중복 호출 방지 큐
- 재발급 실패 시 로그아웃 분기

## 왜 이렇게 했는가

interceptor에 로직을 모은 이유 — 호출부에 분기 코드를 흩지 않기 위함. 트레이드오프는 interceptor 자체 복잡도 증가, 단위 테스트로 보완.

## 진입점

`src/api/interceptor.ts:42-120` 부터.

## 테스트 목록(todo)

- [x] 토큰 만료 시 자동 재발급 후 원 요청 성공
- [x] 동시 다중 요청에서 재발급은 1회만 수행
- [x] 재발급 실패 시 로그아웃
- [ ] (수동) iOS 17 디바이스에서 30분 백그라운드 후 진입 시나리오
```

After는 독자를 동기 → 골격 → 의사결정 → 진입점 → 검증 순서로 안내한다. 본유적 부하가 의도된 방향(이해)으로 흐른다.

### 예시 4: 줄글로 풀어쓴 plan vs 4줄 블록 단위

**Before**

```md
인터페이스를 PaymentProvider로 추출하고 카카오페이와 네이버페이를 각각 구현한 다음 기존 호출부를 마이그레이션하고 구버전을 제거합니다. 테스트는 단위 테스트로 보완합니다.
```

한 단락에 4개의 단계가 압축돼 있다. 독자는 어디서 어느 파일을 봐야 할지 추론해야 한다.

**After**

```md
### 1) PaymentProvider 인터페이스 정의

- **목적**: 결제 모듈을 다중 PG 대응이 가능하도록 표준화한다.
- **변경 파일·함수**: `src/payment/provider.ts`(`PaymentProvider`)
- **핵심 결정**: 동기 메서드 + Promise 반환 — 외부 PG SDK 다수가 비동기라서.
- **검증**: 타입 빌드 통과, 호출부는 다음 단계.

### 2) 카카오페이 PaymentProvider 구현

- **목적**: 인터페이스의 첫 구체 구현.
- **변경 파일·함수**: `src/payment/kakao.ts`(`KakaoProvider`)
- **핵심 결정**: 폴링 대신 웹뷰 콜백 사용 — iOS 17 호환성.
- **검증**: 샌드박스 결제 시나리오 통과.
```

각 단위가 한 블록에 묶여 한 화면에서 매핑이 끝난다. 분할 주의(split-attention)가 사라진다.

## 자주 발생하는 안티 패턴

| 안티 패턴 | 인지 부하 영향 | 대응 |
|---|---|---|
| **전부 한 번에 한 글** (기능+리팩토링+포매팅) | 내재적 부하 폭증, 검토 회피 | 단일 관심사 원칙, 글 체인으로 분할 |
| **포매팅과 로직 혼재** | 외재적 부하 — 의미 있는 변경 찾기 어려움 | 포매팅 단위 선행 분리 |
| **동기(왜) 누락** | 본유적 부하가 추론으로 낭비 | "작업 목적" 섹션을 항상 작성 |
| **도메인 약어 남발** | 사전지식 격차로 부하 증폭 | 첫 등장 시 1줄 설명 |
| **TL;DR 누락된 거대 본문** | 작업기억 초과 | 3줄 요약을 본문 최상단에 배치 |
| **스크린샷 없는 UI 변경** | 외재적 부하 — 코드 읽고 UI 상상 필요 | Before/After 스크린샷·녹화 필수 |
| **검증 항목 없음** | 검증 범위 불명, 본유적 부하 낭비 | "테스트 목록(todo)" 또는 "통과 기준"을 명시 |
| **모호한 제목** (`수정`, `정리`) | 외재적 부하 — 제목으로 분류·우선순위화 불가 | 무엇·어디서·왜를 1줄에 담기 |
| **줄글로만 작성된 단계** | 분할 주의 — 어디부터 볼지 별도 추론 | 한 단위 = 한 블록(목적/진입점/결정/검증) |
| **트리비얼 변경에 풀 양식** | 양식 비용 > 변경 본질 | 조건부 축약(TL;DR + 단계 1~3개 + 검증) |
| **마크다운 양식 파손** | 외재적 부하 — 덩어리 텍스트 | 단락 사이 빈 줄, 헤딩 앞뒤 빈 줄, 코드 블록 언어 명시 |
