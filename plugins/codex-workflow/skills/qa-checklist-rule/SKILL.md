# QA Checklist Rule

## 목적

테스트하는 사람이 별도 판단 없이 순서대로 실행할 수 있는 수동 QA 문서를 만든다. 코드 diff와 관련 파일을 읽은 뒤, 변경된 기능별로 정상 동작과 실패 가능 지점을 함께 체크하게 한다.

## 작성 절차

1. `main` 또는 기준 브랜치 대비 변경 파일을 확인한다.
2. 변경 파일을 기능 단위로 묶는다.
3. 각 기능에서 실제 사용자가 누르는 순서, 권한 요청, 네트워크 실패, 앱 종료/복귀, 저장/삭제/결제/딥링크/비동기 완료 지점을 찾는다.
4. 기능별로 아래 구조를 반복한다.

```md
## N. 기능명

### 기능 테스트

- 확인 항목
  - [ ] 정상동작
  - [ ] 치명적 오류
  - [ ] 사소한 오류
  - 메모:

### iOS 주의

- 확인 항목
  - [ ] 정상동작
  - [ ] 치명적 오류
  - [ ] 사소한 오류
  - 메모:

### Android 주의

- 확인 항목
  - [ ] 정상동작
  - [ ] 치명적 오류
  - [ ] 사소한 오류
  - 메모:

### 다크모드 주의

- 확인 항목
  - [ ] 정상동작
  - [ ] 치명적 오류
  - [ ] 사소한 오류
  - 메모:
```

## 형식 규칙

- Notion에 붙여넣어도 체크박스로 변환되도록 table 안에 `[ ]`를 넣지 않는다.
- 체크박스는 반드시 bullet 하위에 `- [ ]` 형식으로 쓴다.
- 표는 사용하지 않는다.
- 상단 공통 기준은 최소화한다. 각 기능 안에서 바로 체크할 수 있게 적는다.
- 추상 문구만 쓰지 않는다. "언제 네트워크를 끊는지", "어떤 버튼을 누르는지", "무엇이 보이면 정상인지"를 문장에 포함한다.
- 각 확인 항목에는 `정상동작`, `치명적 오류`, `사소한 오류`, `메모:`를 붙인다.

## 오류 등급 기준

- 정상동작: 기대 결과와 다르게 동작한 부분이 없다.
- 치명적 오류: 앱 크래시, 진행 불가, 로그인/결제/통화 실패, 데이터 손실, 잘못된 저장/삭제/구매 처리, 복구 불가 상태.
- 사소한 오류: 문구, 정렬, 색상, 아이콘, 애니메이션, 1회성 지연처럼 우회 가능하고 핵심 기능은 완료되는 문제.

## 필수 검증 축

- iOS 실기기와 Android 실기기를 분리한다.
- Android는 OS 버전별 NativeWind/CSS class 지원 차이를 반드시 확인한다.
- 다크모드는 각 기능 안에 별도 섹션으로 넣는다.
- 네트워크 끊김은 기능별 실행 시점에 붙인다.
- 권한은 권한이 실제로 뜨는 기능에 붙인다.
- 앱 삭제 후 재설치, 기존 설치 업데이트, 앱 종료/복귀는 관련 기능에 붙인다.
- 결제 환경 변수와 sandbox 상품 검증은 결제 기능에 붙인다.
- 비동기 작업은 조기 실패 판정을 피하고 완료/실패/복원 상태를 확인하게 쓴다.
